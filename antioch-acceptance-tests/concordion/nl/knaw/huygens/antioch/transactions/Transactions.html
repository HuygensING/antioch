1) What is the try-confirm-cancel pattern?
The try-confirm-cancel pattern dictates we first confirm any newly created resources before using them.

Failure to follow this pattern leads to grief when operating on the resource!

While the object is TENTATIVE, it cannot be used properly yet, demonstrated by the following request.
When I /resources/73e7a031-8f5b-427d-a39e-7787b97fb9cd/annotations an example annotation body: 
{
  "annotation" : {
    "type" : "t",
    "value" : "v"
  }
}
Then I will get status 409 Conflict and a response body containing: 
{
  "error" : {
    "message" : "This resource has state TENTATIVE; it needs to be CONFIRMED first."
  }

In order to properly use the newly created object, we need to confirm it as follows.
We /resources/73e7a031-8f5b-427d-a39e-7787b97fb9cd/state the body 
{
  "state" : "CONFIRMED"
}
to which the server responds with 204 No Content

Our resource 73e7a031-8f5b-427d-a39e-7787b97fb9cd is now in proper working condition.
To verify we /resources/73e7a031-8f5b-427d-a39e-7787b97fb9cd and observe its CONFIRMED state 
{
  "subresource" : {
    "id" : "73e7a031-8f5b-427d-a39e-7787b97fb9cd",
    "sub" : "/some/folia/path",
    "state" : {
      "value" : "CONFIRMED",
      "since" : "{date.beforeNow}"
    }
  }
}
Now we can resubmit our annotation request.
